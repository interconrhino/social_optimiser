# Recommendation — In Plain Language

**Session:** 0004
**For:** Maria Alcantara and the Student Success team, Rio Valle Community College Network

## 1. What we're recommending, in one paragraph

You need a defensible way to turn a plain-English priority paragraph plus this week's data into a concrete list of about 210 students to call — and you need to show your board, funders, and equity committee how each clause of that paragraph was honoured, and catch unintended side-effects before you deploy. We recommend building a **trade-off calculator**: a small piece of software that takes your paragraph and your current student data, and produces not one answer but a **menu of answers** — each a different weekly call list, each with a scorecard showing exactly how well it satisfies every clause of the paragraph and how coverage of *every* demographic group compares to last semester's baseline. Leadership picks from the menu. The "menu" is generated by a technique called **multi-objective optimisation** — specifically, an approach that sweeps through trade-offs between the paragraph's clauses and returns the best possible call list for each trade-off. You can run it in three modes: "balance all clauses evenly", "protect the clause that would otherwise be worst served", or "maximise retention while keeping all clauses above their floors." All three modes produce the same kind of scorecard, so the board sees the same picture whichever mode is in use.

---

## 2. How it works, step by step

1. **Once per semester**, you sit down with leadership and translate the priority paragraph into a small set of formal clauses — each with a group (e.g., "first-generation students"), a direction (increase, protect, maintain), and a target (e.g., "at least 40% of calls," or "not below last semester's level"). This is a facilitated workshop, explicitly documented, so there is a paper trail from "what leadership wrote" to "what the system optimises for."
2. **Once**, a data scientist builds a small predictive model that estimates, for any student in any week, the expected gain in end-of-semester retention if that student gets a call this week. This uses your three years of historical data, with a statistical correction for the fact that past calls were not randomly assigned (a technique called inverse-propensity weighting — it adjusts for the bias in the historical data).
3. **Every Monday morning**, the system takes this week's watch-list (~2,100 students), runs the predictive model to score each one, and solves a small mathematical puzzle: pick 210 students to call, respecting the weekly budget and per-campus caps, that performs best on the chosen trade-off mode.
4. **The output** is a proposed call list plus a scorecard that shows:
   - Predicted retention benefit from this list (the headline number).
   - For each clause of the paragraph: how well this list honours it (e.g., "first-gen coverage: 45% — target was ≥40%, green"; "adult-learner coverage: 19% — last-semester baseline was 18%, green").
   - For every monitored demographic slice, including ones *not* named in the paragraph: coverage vs. last semester (with red flags if a slice has quietly dropped more than a set threshold — this is the 30%-drop-to-students-without-dependants early-warning system).
5. **Leadership reviews** the scorecard and either approves the list or asks for a different trade-off (e.g., "show me what it looks like if we protect adult learners harder"). The system re-runs in under a minute.
6. **Coaches make the calls**. Outcomes feed back into the data, and the predictive model is refreshed between semesters.
7. **At the board meeting**, the scorecard is the audit trail: "here is the paragraph you wrote, here is how it was turned into clauses, here is the Pareto menu we considered, here is the list we chose, here is how every named group and every monitored group scored."

## 4. What you will get — concrete deliverables

- A **weekly call list** of ~210 students, respecting all capacities and clause floors, produced in under a minute.
- A **per-list scorecard** showing retention, per-clause satisfaction, and per-monitored-slice coverage (including the red-flag early warning for unnamed slices).
- A **Pareto menu** (optional, run whenever leadership wants to explore): several alternative call lists, each reflecting a different trade-off between retention and clause satisfaction, displayed as a chart so the trade-offs are visible at a glance.
- A **board-ready audit trail** for every semester: paragraph → clauses → Pareto menu → chosen list → realised scorecard at end of semester.
- A **side-effect alarm system**: any recipe that would quietly drop coverage of any monitored group by more than a leadership-set threshold is flagged before deployment.

## 5. Confidence and how you will know it is working

- **Mathematical confidence**: for any fixed trade-off, the call list is *provably optimal* — the software is guaranteed to return the best possible list under that trade-off. No hidden shortcuts, no evolutionary-algorithm guesswork. That is what makes this defensible to your equity committee.
- **Empirical confidence**: the predictive model that estimates per-student retention benefit is validated against your 3-year history using standard cross-validation. You will see calibration plots (is a "predicted +5% retention" actually +5% on held-out data?). If calibration is poor, that is diagnosed up front, not hidden.
- **How to know it is working in the field**:
  - Compare end-of-semester retention to last semester, matched on student profile.
  - Track per-clause satisfaction every semester; the scorecard will make it obvious if any clause is drifting away from target.
  - Audit side-effect alarms: did the system warn you about any unintended drops, and were those warnings accurate?
  - Track coach override rate: if coaches override the list frequently, their reasons are a signal the model is missing something (and feed back into improvement).

## 6. Next steps — a pragmatic roadmap

**Weeks 1–2**: Formalise the last-semester baseline coverage numbers, confirm per-campus capacity split, and convene leadership for a first paragraph-to-clause translation workshop.

**Weeks 3–4**: Build the retention-benefit predictive model on your 3-year history with the off-policy correction. Validate with cross-validation; present calibration curves to leadership.

**Weeks 5–6**: Build the optimiser. A competent data scientist or OR-trained analyst can write this in Python using `Pyomo` (modelling) and `HiGHS` (free, open-source solver) or Gurobi (free for academics). The code to run the three modes (Pareto sweep / weighted goal programming / max-min) is a few hundred lines.

**Weeks 7–10**: Pilot on one campus — the one with the cleanest data and the most engaged director. Run as a shadow system for 4 weeks (coaches make the real decisions; optimiser just produces recommendations), then switch to optimiser-first with coach override for the remaining 8 weeks.

**After the pilot**: review results with the board and equity committee (the scorecards and the Pareto menu are your primary artefacts). If the per-clause satisfaction story holds and retention is preserved, roll out to all four campuses next semester.

**Ongoing**: refresh the predictive model between semesters; refresh the paragraph-to-clause translation at the start of each semester; refresh the per-slice baseline annually.

Realistic timeline from "go" to "pilot running on one campus": **about 6–10 weeks** of concentrated effort from one data scientist and one analyst, plus a few hours from leadership for the workshop and sign-offs.

## 7. What to watch out for (honest caveats)

- **The predictive model is the quietest and most consequential piece.** The optimiser trusts it completely. If it is mis-calibrated — particularly due to uncorrected bias in the historical call log — the system will confidently recommend the wrong students. Invest real time in the off-policy correction and calibration checks, and be prepared to iterate. If in doubt, start conservative: use the model's *rank* (who is more likely to benefit) rather than its *absolute numbers*.
- **"The paragraph" is a political artefact as much as a technical one.** The quality of the output depends on an honest, careful translation from leadership's words into formal clauses. Do not let a data scientist do this alone — it is a facilitated workshop, and it should produce a short document leadership signs off on.
- **This approach does not explicitly plan across the 12 weeks.** It re-optimises each Monday using fresh data. In most cases that is fine and is what makes it auditable, but it means the system can't explicitly reason "let me not call student A this week so I have capacity to call them next week when they'll be in worse shape." A sequential-planning upgrade is possible later if the simpler approach leaves meaningful value on the table — but start simple.
- **Peer effects between students are not modelled.** If students in the same programme, dorm, or cohort influence each other's engagement, your system is leaving that information out. Worth flagging to leadership as a "known unknown" and potentially modelling in a later iteration.
- **Clause growth risk.** The trade-off sweep gets exponentially more expensive as the paragraph grows in clauses. Keep the paragraph under ~6 clauses; if it must grow, the goal-programming mode (single-IP with weighted deviations) scales cleanly.
- **Coach buy-in**: the list is a recommendation, and coaches will know students the model does not. Build in an easy override path *and* log every override with a reason — the overrides become valuable data for improving the model and for spotting gaps between the engagement proxy and lived reality.
